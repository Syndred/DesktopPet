# 《轻宠·领主 3D》任务文档（Execution Tasks）

文档版本：v1.1  
创建日期：2026-03-04  
对应需求文档：`docs/requirements.md`  
说明：本版本仅做任务编排，不绑定具体排期时间。

## 1. 执行原则

1. 先闭环后扩展：先交付“收留 -> 对战 -> 占领 -> 收益反哺”的完整链路。
2. 先可观测后优化：关键事件埋点、日志、报警要与功能同步落地。
3. 先正确再性能：先保证规则与数据一致性，再做性能压缩。
4. 每个任务必须具备：产出物、测试要求、验收标准。

## 2. 最新性能基线（放宽版）

1. PC 内存：`p95 < 150MB`（单宠物实例，空闲场景）。
2. 移动端内存：`p95 < 220MB`（Flutter + Unity 前台场景）。
3. API 响应：`p95 < 100ms`。
4. 对战同步：`p95 < 150ms`。
5. 崩溃率：`< 0.1%`。

## 3. 任务分组总览

1. G0 工程与数据基础。
2. G1 核心闭环能力。
3. G2 体验与增长能力。
4. G3 测试、发布与运维。

## 4. 详细任务清单（无时间版）

### G0 工程与数据基础

T-001 Monorepo 与基础工程初始化  
目标：建立统一工程结构、规范和基础 CI。  
前置依赖：无。  
主要产出：
1. `apps/`、`packages/` 基础目录。
2. lint、format、type-check、commit 规范。
3. 一键启动脚本与环境变量模板。  
测试要求：
1. 执行 lint、type-check、基础单测，结果全绿。
2. 验证新成员按 README 可在新环境启动。  
验收标准：
1. 任意端可本地启动成功。
2. CI 至少包含代码风格、类型检查、单测 3 条流水线。

T-002 共享协议与错误码体系  
目标：统一跨端类型、接口契约、错误处理。  
前置依赖：T-001。  
主要产出：
1. `packages/shared-types` 的 DTO、事件、枚举。
2. 错误码定义（鉴权、业务、系统）与说明文档。  
测试要求：
1. 契约测试：PC/Mobile/Edge 使用同一版本类型可编译通过。
2. 错误码映射测试：关键错误码可正确被前端展示。  
验收标准：
1. 不再出现跨端字段命名不一致问题。
2. 关键接口返回格式可被自动化测试校验。

T-003 Supabase 数据层与 RLS  
目标：建立核心表结构、RLS、索引与迁移机制。  
前置依赖：T-001、T-002。  
主要产出：
1. 数据库迁移脚本（用户、宠物、点位、对战、领地等）。
2. PostGIS 扩展与地理索引。
3. RLS 策略与权限矩阵文档。  
测试要求：
1. SQL 单元测试：核心查询与写入脚本可重复执行。
2. 安全测试：未授权访问被拒绝，越权写入被拒绝。  
验收标准：
1. 全量迁移在空库可一次性执行成功。
2. 关键查询在测试数据量下满足目标延迟。

T-004 埋点、日志与报警基础  
目标：形成最小可观测能力。  
前置依赖：T-001、T-002。  
主要产出：
1. 统一事件命名与埋点 SDK 包装层。
2. 客户端错误日志上报与后端异常告警规则。
3. 基础看板（闭环转化、崩溃、延迟）。  
测试要求：
1. 埋点联调测试：关键事件可在看板中查询到。
2. 告警演练测试：模拟故障可触发告警。  
验收标准：
1. PRD 定义的关键事件覆盖率达到 100%。
2. 故障发生后 5 分钟内可定位到模块级问题。

### G1 核心闭环能力

T-101 PC 透明窗口、托盘与窗口控制  
目标：实现桌面常驻宠物窗口的基础能力。  
前置依赖：T-001、T-002。  
主要产出：
1. Electron 透明无边框窗口。
2. 托盘菜单（显示/隐藏/退出）与开机自启。
3. 拖拽、缩放、吸边、位置持久化。  
测试要求：
1. 功能测试：窗口行为在多分辨率、多缩放下稳定。
2. 兼容测试：全屏应用场景不抢焦点。  
验收标准：
1. 窗口操作无明显抖动、黑边、闪烁。
2. 位置与窗口状态可跨重启恢复。

T-102 模型命中检测、鼠标穿透与 R3F 渲染  
目标：实现“模型可交互、透明区域可穿透”。  
前置依赖：T-101。  
主要产出：
1. 命中检测（像素/包围盒）方案。
2. 动态 `setIgnoreMouseEvents` 切换逻辑。
3. Idle/Attack/Hit/Dodge 动画状态与降帧策略。  
测试要求：
1. 自动化交互回放测试：透明区域误拦截统计。
2. 性能测试：静止与交互场景 FPS/CPU/内存采样。  
验收标准：
1. 透明区域误拦截率 < 1%。
2. PC 性能达到放宽后生死线。

T-103 `BattleEngine.ts` 回合制规则引擎  
目标：实现五行克制、博弈三角、异常状态、怒气机制。  
前置依赖：T-002。  
主要产出：
1. `packages/game-engine/BattleEngine.ts`
2. 规则文档与配置化参数。
3. 回合结果结构化战报。  
测试要求：
1. 单元测试：覆盖克制链、异常、怒气、边界值。
2. 属性测试：随机输入下不出现非法状态。  
验收标准：
1. 规则单测通过率 100%。
2. 关键公式与 PRD 规则零偏差。

T-104 实时对战房间、同步与反作弊  
目标：完成 1v1 对战的网络与校验链路。  
前置依赖：T-003、T-103。  
主要产出：
1. 房间创建/加入/重连机制。
2. Realtime 回合广播与状态恢复。
3. 边缘函数合法性校验与审计日志。  
测试要求：
1. 集成测试：双端对战全流程。
2. 弱网测试：抖动、丢包、断网重连。
3. 攻击测试：伪造指令、越权房间写入。  
验收标准：
1. 对战同步延迟 p95 < 150ms。
2. 非法指令拦截率（在攻击脚本覆盖范围）= 100%。

T-105 LBS 点位刷新与地理围栏  
目标：完成流浪宠物刷新与 100m 收留触发。  
前置依赖：T-003。  
主要产出：
1. 点位刷新调度逻辑（地标映射、稀有度、时段）。
2. 围栏判定与距离误差补偿。
3. 漂移过滤策略。  
测试要求：
1. 地理仿真测试：不同场景轨迹回放。
2. 精度测试：室内外与弱 GPS 条件下误差统计。  
验收标准：
1. 触发误差 p95 < 50m。
2. 地标与属性映射准确率 100%。

T-106 收留、图鉴与记忆标签  
目标：形成收留结果的可追溯资产。  
前置依赖：T-105。  
主要产出：
1. 收留事务流程（写入宠物、图鉴、标签）。
2. 图鉴查询 API 与标签回显接口。  
测试要求：
1. 事务一致性测试：中断场景下回滚或补偿正确。
2. 接口测试：多端读取一致性。  
验收标准：
1. 收留成功后宠物、图鉴、标签数据一致率 100%。
2. 重启或重登后标签可完整恢复。

T-107 情绪镜像引擎（低打扰）  
目标：把行为统计映射为温和反馈动作。  
前置依赖：T-002、T-004。  
主要产出：
1. 行为统计采集（不采集正文）。
2. 情绪映射规则与节流机制。
3. 每日情绪小结生成。  
测试要求：
1. 隐私测试：确认无敏感正文采集。
2. 规则测试：不同行为输入触发正确反馈。
3. 体验测试：高频使用下不出现连续打扰。  
验收标准：
1. 数据采集合规通过审计。
2. 情绪反馈触发准确且无明显扰民行为。

T-108 区域占领最小可用版  
目标：打通入侵、驻守、占领、收益翻倍。  
前置依赖：T-104、T-106。  
主要产出：
1. 区域归属状态机。
2. 占领奖励与收益系数计算。
3. 占领记录流水。  
测试要求：
1. 规则测试：胜负与占领归属一致性。
2. 并发测试：同区域并发入侵冲突处理。  
验收标准：
1. 占领状态变更正确率 100%。
2. 收益加成实时生效，跨端同步延迟 < 200ms。

T-109 核心闭环联调  
目标：联通“收留 -> 对战 -> 占领 -> 收益反哺”全链路。  
前置依赖：T-102、T-104、T-106、T-108。  
主要产出：
1. 闭环联调版本。
2. 关键链路回滚与补偿机制。  
测试要求：
1. E2E 测试：完整闭环自动化脚本。
2. 稳定性测试：长时间运行与多次重复闭环。  
验收标准：
1. 闭环通过率 >= 98%。
2. 无 P0/P1 阻断问题。

### G2 体验与增长能力

T-201 轻社交（微光相遇 + 匿名守护）  
目标：在不增加社交压力的前提下增加陪伴温度。  
前置依赖：T-106、T-109。  
主要产出：
1. 同点位宠物自动互动。
2. 匿名守护反馈流程。  
测试要求：
1. 隐私测试：无法反查守护者身份。
2. 功能测试：同点位触发稳定性。  
验收标准：
1. 互动触发成功率 >= 99%。
2. 匿名链路无身份泄露风险。

T-202 好友与私密约战  
目标：提供轻量熟人社交对战入口。  
前置依赖：T-104。  
主要产出：
1. 好友关系与邀请流程。
2. 私密房间约战。  
测试要求：
1. 功能测试：邀请、接受、开战、结算全流程。
2. 安全测试：未授权用户不可加入私密房间。  
验收标准：
1. 私密房间流程成功率 >= 99%。
2. 越权加入拦截率 100%。

T-203 养成系统（等级、亲密度、技能）  
目标：提升长期留存与成长目标感。  
前置依赖：T-103、T-106。

2. 技能解锁与升级。

测试要求：
1. 规则测试：成长曲线与阈值正确。
2. 回归测试：成长后战斗计算不破坏平衡。  
验收标准：
1. 成长数据与战斗表现一致。
2. 不出现负成长、越级解锁等非法状态。

T-204 外观与换装  
目标：提供可视化个性化能力。  
前置依赖：T-102、T-203。  
主要产出：
1. 外观资源管理与切换机制。

测试要求：
1. 渲染测试：不同外观组合无穿模/丢材质。
2. 兼容测试：不同设备加载稳定性。  
验收标准：
1. 换装成功率 >= 99%。

## 6. 质量门禁（全任务通用）

代码门禁：
1. lint/type-check/test 必须全绿。
2. 核心模块变更必须附带测试用例。

功能门禁：
1. P0/P1 不允许进入发布候选版本。
2. 需求项必须可追溯到任务、测试、验收记录。

性能门禁：
1. 不得破坏放宽后生死线指标。
2. 每次性能优化必须给出前后对比数据。

安全门禁：
1. RLS 策略变更必须二次审查。
2. 关键写操作必须经服务端校验。
